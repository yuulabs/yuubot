1 Ingress Rule 是什么？

Ingress Rule（入口规则）是 yuubot 中的消息路由声明。 它决定了来自哪个集成（Integration）的、什么类型的消息， 应该被投递到哪个 Actor 的邮箱中。

简单说：「来自 Slack 的消息 → 发给客服助手」、 「来自 GitHub 的 PR 事件 → 发给代码审查 Agent」。

💡 系统消息不经过 Ingress Rule
Admin Conversation、定时任务、后台回调等系统消息会通过专用通道直接投递到 Actor 邮箱，不走 Ingress Rule 的路由匹配。前端 Routes 页面也只展示外部 Integration 来源的规则。

2 字段详解

source_id_pattern

匹配发送消息的集成（Integration）ID。

怎么知道 source_id 是什么？
Source ID 就是集成创建时指定的 ID。去{" "} Integrations 管理页 查看已有的集成，它们的 ID 就是 message source 的 ID。
示例：slack-main → 只匹配 Slack 主工作区
telegram-* → 匹配所有 Telegram 集成
* → 匹配所有集成

source_path_pattern

匹配集成内部的路径（频道、仓库、群组等）。

示例：channels/dev → 只匹配 #dev 频道
channels/* → 匹配所有频道
repos/** → 匹配所有仓库及子路径

kind_patterns

过滤消息的种类。多个种类用逗号分隔。

示例：mention → 仅 @提及
dm → 仅私聊
text, image → 文本或图片消息

actor_id

目标 Actor 的 ID。匹配的消息会发到该 Actor 的邮箱。

一个消息可以被多个规则匹配（扇出）， 同时投递到多个 Actor。
Glob 模式速查表
模式含义示例匹配
*匹配单个路径段中的任意字符slack-* → slack-main
**匹配任意层级路径（含空）repos/** → repos/org/repo
?匹配单个字符channel? → channel1
[abc]匹配字符集ch[12] → ch1

多个规则同时匹配

同一个来源的消息可以被多条 Ingress Rule 同时匹配。 如果两个人分别创建了：

Source IDPathKinds→ Actor
slack-mainchannels/support*support-agent
slack-mainchannels/support*logger-agent

那么一条来自 Slack 客服频道的消息会同时发送到 support-agent 和 logger-agent 两个 Actor 的邮箱。这是实现扇出（fan-out）的典型方式。

创建规则的建议

  1. 从具体到通用 — 先创建精确匹配的规则，再考虑通配符。
    例如先用 slack-main / channels/dev / mention，确认工作正常后再放宽。
  2. 仅创建需要的规则 — 每条规则在消息到达时都会被评估。
    不必要的宽泛规则可能导致消息发给错误的 Actor。
  3. 利用 Kind 做精细控制 — 如果只想处理 @提及 而非所有消息，设置 kind_patterns = mention。
  4. 记得先创建 Actor 和 Integration — 规则引用的 Actor 和 Integration 需要先存在。

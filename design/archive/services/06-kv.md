# Design: Actor JSON KV

**实现顺序：6**（依赖 [02-admin-boundary.md](02-admin-boundary.md)）

## Scenario

### Motivation

KV 为 LLM 在管理面构建**可读写状态的动态网页**（尤其可提交表单）提供 actor 命名空间 JSON
存储。主要消费方是 LLM 写入 workspace 的 HTML/CSS/JS：管理员经 browse/preview 打开后，页面
JS 经 AdminAuth 调用 KV HTTP 持久化/读取 JSON（草稿、多步表单、配置等）。

与 [03-inbound.md](03-inbound.md) 的 actor inbound 组合，可形成应用层闭环：

```text
LLM 写 HTML → 用户在管理面打开并交互 → 提交时 POST inbound → 唤醒 LLM 继续处理
```

URL 公式见 [deployment-design.md](../deployment/deployment-design.md)（`kv_url`、
`actor_inbound_url`）。KV 仅在 `admin_url_base`；公网 Share 为只读静态快照，**不得**访问
KV（见 [05-share.md](05-share.md)）。KV 与 inbound 无模块硬耦合，组合方式同 README 中 tasks +
inbound 先例。

### Typical trace

1. LLM 将 `projects/.../form.html` 写入 workspace，内嵌 `fetch(kv_url)` 与提交 handler。
2. 管理员在管理面打开页面；JS 经 AdminAuth 读写 KV。
3. 用户提交 → `PUT` KV（可选，存完整状态）+ `POST` inbound（`text` 含时间、来源页、业务上下文与
   `payload`；`conversation_id` 对齐当前会话）。
4. LLM 被唤醒，从 inbound `text` 获知提交语境；必要时再读 `kv_key` 或 workspace 文件继续处理。

### Access paths

| 调用方 | 入口 | 说明 |
| --- | --- | --- |
| LLM 编写的 HTML/JS | `GET` / `PUT` / `DELETE` KV HTTP | 浏览器带 AdminAuth；见下节 HTTP contract |
| 管理员 UI（可选） | 同上 | 与 HTML 共享端点 |
| LLM（`execute_python`） | 无 KV facade | 通过 inbound 载荷或 workspace 文件获取提交结果；prompt 说明动态页模式 |

## Concepts

```text
JsonDocument = actor 命名空间下的一个 JSON 值
KvStore      = Runtime.kv
Key          = posix 相对路径段，无 ".."、无 leading /
```

磁盘：`data_dir/kv/{actor_id}/{key_path}.json`。`key_path` 为规范化后的 key，其中 `/` 映射为
子目录（见 Key 规则）。

## Shapes

```py
class JsonDocument(msgspec.Struct, frozen=True):
  actor_id: str
  key: str
  value: object
  updated_at: str      # ISO8601 UTC
  etag: str            # 内容哈希，见 ETag

class KvStore:
  async def get(self, actor_id: str, key: str) -> JsonDocument | None: ...
  async def put(
    self, actor_id: str, key: str, value: object, *, if_match: str | None = None,
  ) -> JsonDocument: ...
  async def delete(self, actor_id: str, key: str) -> bool: ...
```

## Key 规则

| 规则 | 说明 |
| --- | --- |
| 规范化 | 拒绝 `..`、绝对路径、`\`；连续 `/` 合并 |
| 磁盘映射 | `key="foo/bar"` → `kv/{actor_id}/foo/bar.json` |
| 目录 vs 文件冲突 | 若存在目录 `kv/{actor_id}/foo/`（因 key `foo/baz` 创建），则 key `foo` **不可** PUT；返回 `409 conflict` `code=conflict`, `detail.reason=key_collides_with_prefix` |
| 反向冲突 | 已存在 `foo.json` 时，PUT `foo/bar` → `409` |

## Value 与 ETag

| 项 | v1 |
| --- | --- |
| JSON size limit | 单文档 **1 MiB** 序列化后；超限 `400 bad_request` |
| `etag` | `sha256(canonical_json(value))` 十六进制；`canonical_json` = `json.dumps(sort_keys=True, separators=(',',':'))` |
| 并发写 | `put` 对单文件 `write temp + os.replace`；`if_match` 不匹配 → `409 conflict` |
| 无 if_match | 最后写 wins（仍原子替换文件） |

## Core flow

```py
async def put(actor_id, key, value, if_match=None):
  normalize_key(key)
  assert_json_size(value, max_bytes=1_048_576)
  path = kv_path(actor_id, key)
  assert_no_prefix_collision(actor_id, key)
  if if_match and read_etag(path) != if_match:
    raise Conflict()
  new_etag = compute_etag(value)
  write_atomic(path, value)   # .tmp + replace
  return JsonDocument(actor_id=actor_id, key=key, value=value, updated_at=now(), etag=new_etag)
```

## HTTP（admin only）

错误信封见 [02-admin-boundary.md](02-admin-boundary.md)。路径：
`/api/actors/{actor_id}/kv/{key}`，`{key}` 为 URL 编码的 posix key（`foo/bar` → `foo%2Fbar`）。

### `GET`

Success `200`:

```json
{
  "actor_id": "amy",
  "key": "settings",
  "value": { "theme": "dark" },
  "updated_at": "2026-07-03T09:00:00Z",
  "etag": "a1b2..."
}
```

| Status | code | 场景 |
| --- | --- | --- |
| 404 | `not_found` | key 不存在 |
| 400 | `bad_request` | key 非法 |

响应头：`ETag: "{etag}"`（带引号）。

### `PUT`

Request:

```json
{ "value": { "theme": "light" } }
```

可选头：`If-Match: "{etag}"`。

Success `200` — 同 GET body（新 `etag` / `updated_at`）。

| Status | code | 场景 |
| --- | --- | --- |
| 400 | `bad_request` | value 非 JSON 对象/数组、超大 |
| 409 | `conflict` | etag 不匹配或 key 前缀冲突 |
| 404 | `not_found` | 未知 `actor_id` |

### `DELETE`

Success `200`:

```json
{ "actor_id": "amy", "key": "settings", "deleted": true }
```

不存在 `404 not_found`。幂等：可选 v1 对不存在也返回 `200 deleted: false`；推荐 **404**。

## Prompt guidance

Actor system prompt 与 `execute_python` tool description 应告知 LLM 可使用动态页模式（参照
[04-tasks.md](04-tasks.md) 对 `yb.tasks` 的说明方式）。**不设 Python facade**；交互由浏览器侧
HTML 完成。

须告知的内容：

- 可向 workspace 写入 HTML/CSS/JS；在管理面打开后，页面 JS 可调用：
  - `GET` / `PUT` / `DELETE` `/api/actors/{actor_id}/kv/{key}`（`{key}` URL 编码；支持
    `ETag` / `If-Match`）
  - `POST` `/api/actors/{actor_id}/inbound`（`text` + 可选 `conversation_id`）
- 推荐模式：提交时 inbound 除 payload 外须携带 LLM 编写页面时已知的上下文，使唤醒后无需猜测来源。
  `text` 建议为结构化 JSON 字符串（或等价可读格式），至少包含：
  - `submitted_at`：ISO8601 时间
  - `source_page`：workspace 相对路径（如 `projects/q3-review/form.html`）
  - `purpose` / `context`：LLM 写页时嵌入的说明（表单用途、关联任务、业务 id 等）
  - `kv_key`（若使用 KV）：对应存储 key，便于 LLM 定位持久化状态
  - `payload`：用户本次提交的数据
  可选 `source` 字段放机器侧元数据（`kind: "html_form_submit"`、`page`、`kv_key`），与
  [03-inbound.md](03-inbound.md) 的 `source` 字典一致。
- **禁止**在 `execute_python` 内直接 loopback 调 admin HTTP（与 tasks 提示一致）。

实现落点（供后续 PR 引用）：

- `src/yuubot/actor/prompt.py` — `_tool_suggestions()` 或独立小节
- `src/yuubot/tools/execute_python.py` — `DESCRIPTION` 补充动态页能力

组合示例（页面内 JS）。`PAGE_META` 由 LLM 写 HTML 时嵌入，描述该页身份与业务上下文：

```js
// LLM writes these constants when generating the page.
const PAGE_META = {
  source_page: "projects/q3-review/approval-form.html",
  purpose: "Q3 budget line-item approval for conv c1",
  kv_key: "forms/q3-approval/draft",
  conversation_id: "c1",
};
const actorId = "amy";

async function loadDraft() {
  const res = await fetch(
    `/api/actors/${actorId}/kv/${encodeURIComponent(PAGE_META.kv_key)}`,
  );
  if (res.ok) return (await res.json()).value;
  return {};
}

async function submitForm(payload) {
  await fetch(
    `/api/actors/${actorId}/kv/${encodeURIComponent(PAGE_META.kv_key)}`,
    {
      method: "PUT",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ value: payload }),
    },
  );

  const inboundText = JSON.stringify({
    kind: "html_form_submit",
    submitted_at: new Date().toISOString(),
    source_page: PAGE_META.source_page,
    purpose: PAGE_META.purpose,
    kv_key: PAGE_META.kv_key,
    payload,
  });

  await fetch(`/api/actors/${actorId}/inbound`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      text: inboundText,
      conversation_id: PAGE_META.conversation_id,
      source: {
        kind: "html_form_submit",
        page: PAGE_META.source_page,
        kv_key: PAGE_META.kv_key,
      },
    }),
  });
}
```

唤醒后 LLM 从 `text` 即可得知「何时、哪张页、为何、提交了什么」；需要完整历史状态时再读
`kv_key` 对应文档。

## Context access

```text
Core needs: actor_id, key, value
Source:
  HTTP <- Facade after AdminAuth -> runtime.kv
```

## Invariants

1. 无公开 KV 路由。
2. Key 规范化后唯一对应磁盘路径或明确 `409` 冲突。
3. 单文档 ≤ 1 MiB；原子 replace 写入。
4. HTTP `ETag` / `If-Match` 与 `JsonDocument.etag` 一致。

# yuubot 真实需求

本文档描述当前这一轮重设计真正要满足的需求。它不是历史设计总结，也不是实现笔记。后续所有设计和分工都以本文档为准。

## 1. 系统要解决什么问题

yuubot 是一个消息驱动的 agent 宿主。

它接收 IM 消息，决定是否需要响应；如果需要，就把消息和必要上下文交给 LLM；LLM 再通过 bot 提供的内建能力完成查询、思考和发送消息。

我们现在只关心下面几件事：

1. 消息如何进入系统。
2. 系统如何判断这条消息是命令，还是会话续传。
3. LLM 最终会看到什么输入。
4. LLM 如何调用 bot 的能力。
5. 会话如何延续，而不是每次都重新开始。

除此之外的概念，如果不能直接服务于这些需求，就不应该进入新设计。

## 2. 核心使用方式

### 2.1 用户如何使用 bot

用户和 bot 的交互只有两类：

1. 发命令。
2. 给一个已经开始的会话继续发消息。

典型例子：

1. 群聊里 `@bot /yllm 帮我总结一下这几张图`
2. 私聊里 `/yllm#coder 帮我看这个报错`
3. 会话已经开始后，用户继续补充一句“还有这个日志”
4. 管理员用 `/ybot on`、`/ybot off`、`/yclose` 这类命令控制行为

### 2.1.1 命令前缀语法

命令系统有一个必须明确写清的规则，叫“区分前缀”。

系统支持多个前缀，例如：

1. `y`
2. `yuu`

它们存在的原因是避免和其他 bot 的通用命令撞车，例如裸 `/llm` 很容易冲突。

当前语法规则是：

1. 常用写法是 `y<command>`。
2. 兼容写法也允许 `y <command>`。
3. 命令解析时，前缀部分会先被识别，再被移除，然后对剩余文本做 `strip` 和命令匹配。
4. 因此 `yllm` 和 `y llm` 在语义上都可以进入同一个命令节点。

例子：

1. 正确：`/yllm 帮我总结`
2. 正确：`/y llm 帮我总结`
2. 正确：`/yllm#coder 看这个报错`
3. 正确：`/ybot on`
4. 正确：`/y bot on`

但在真实使用上，仍然应该把下面这件事讲清楚：

1. QQ 里敲空格不方便。
2. 所以常用示例应该优先写成 `y<command>`。
3. 兼容写法存在，但不是主要示例。

这条规则会直接影响命令树和 route 设计，因此后续实现必须保留“区分前缀”这个能力。

### 2.2 LLM 如何使用 bot

yuubot 为 LLM 准备两类东西：

1. tool
2. capability

这里要区分清楚：

1. tool 是更一般的工具能力。
2. capability 是 yuubot 自己提供的一组内建能力。
3. 只有 capability 这一类，才统一通过 `cap_call_cli` 之类的入口，用 raw CLI 风格调用。

我们这一轮设计讨论的重点，是 capability 的调用协议，而不是要把所有 tool 都压成同一种形式。

因此，准确说法应该是：

1. capability 只通过 raw CLI 字符串调用。
2. 不为 capability 再引入第二套协议。

不引入第二种 capability 调用协议，不要求 LLM 为 capability 记 JSON schema，也不为 capability 单独发明新的 DSL。

原因很简单：

1. CLI 紧凑。
2. usage 文档容易写。
3. 人和 LLM 都容易读。
4. token 成本可控。
5. 已有能力天然适合“命令 + 参数”的表达方式。

约束也很简单：

1. LLM 通过 `cap_call_cli` 传 raw CLI。
2. 系统负责解析 raw CLI。
3. 复杂结构数据仍然允许放在 `--` 后面作为原始 payload。

例子：

```text
im send --ctx 12 -- [{"type":"text","text":"你好"}]
mem recall --ctx 12 --query "偏好 猫"
web read https://example.com
```

这里的关键点不是“它看起来像 CLI”，而是：

1. 对 capability 只暴露一种调用格式。
2. 对实现内部保留 typed request 和 typed response 的空间。

## 3. 本轮必须满足的功能

### 3.1 命令树

命令树继续保留。

命令树的职责很单纯：

1. 把输入文本匹配成 route。
2. 决定命中了哪个命令节点。
3. 给出剩余参数文本。

命令树还必须符合真实交互语法：

1. 系统支持“区分前缀”。
2. `y<command>` 是主要写法。
3. `y <command>` 是兼容写法。
4. 命令树匹配时，应该先识别并剥离前缀，再对剩余文本做统一匹配。

命令树不负责：

1. 会话管理。
2. LLM 调度。
3. 消息渲染。
4. 发送回复。

### 3.2 会话续传

系统必须支持多轮会话。

这里的真实需求是：

1. 用户发起一次 LLM 任务后，后续消息可以继续这次任务。
2. 同一个上下文里，系统知道当前是否存在活动中的会话。
3. 会话可以超时、关闭、切换 agent、必要时做摘要续传。
4. 用户不用每次都重新描述整个任务。

我们需要的是“统一的会话续传模型”，而不是零散的：

1. session
2. active flow
3. ping flow
4. auto mode current agent

这些现有概念在新设计里应该尽量合并。

### 3.3 Capability

bot 要向 LLM 暴露一组内建能力，例如：

1. IM
2. Memory
3. Web
4. Image
5. Schedule

新设计统一称之为 capability。

一个 capability 对外只需要表达三件事：

1. 它叫什么。
2. 它有哪些 action。
3. 每个 action 通过 `cap_call_cli` 调用时的 raw CLI usage 是什么。

### 3.4 LLM 输入透明

这是本轮最重要的非功能需求之一。

必须做到：

1. 看一份配置或文档，就知道 LLM 最终会看到怎样的消息。
2. 不需要去多个 Python 文件里追格式化逻辑。
3. 不需要猜 reply、图片、用户名、上下文提示到底怎么渲染。

因为 LLM 输入直接影响账单、效果和调试成本，所以消息渲染必须是显式设计，而不是散落在代码里的隐式行为。

### 3.5 统一的续传方案

现在的 ping flow、队列、continue、auto continuation 本质上都在处理同一件事：

1. 当用户在已有对话中继续发消息时，系统如何把新消息并入当前任务。

新设计必须用一套统一方案描述它，而不是多套局部机制并存。

## 4. 本轮明确不要做什么

### 4.1 不追求保留旧概念

如果旧概念已经不再简洁，就不要硬保留名字和边界。

### 4.2 不把 transport 格式暴露给上层

OneBot 事件、数据库记录、内部渲染文本，不应该混成同一种“消息”。

### 4.3 不把运行时技巧当架构

懒加载、全局注册表、字符串拼接、dict 透传，都只能是局部实现细节，不能成为设计核心。

### 4.4 不为 LLM 设计第二套复杂协议

Capability 的调用仍然保持 raw CLI 形式，不引入 JSON-first tool schema 作为主协议。

## 5. 设计约束

新设计至少要满足这些约束：

1. 关键概念少。
2. 每个概念职责单一。
3. LLM 输入可审计。
4. Capability 调用协议唯一。
5. 会话续传方案统一。
6. 模块之间依赖方向清晰。
7. 支持后续并行分工，不需要一个人一次性重写全部。

## 6. 交付标准

后续设计文档必须能直接回答下面几个问题：

1. 一条消息进入系统后，只会走哪几条固定路径？
2. 谁负责判断命令，谁负责判断会话续传？
3. 谁负责决定 LLM 看见的输入？
4. LLM 通过什么协议调用 capability？
5. capability 的 usage 文档放在哪里，谁维护？
6. 会话、续传、超时、关闭、摘要续传是否属于同一个模型？
7. 哪些部分可以拆给不同的人并行实现？

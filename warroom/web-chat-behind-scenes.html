Web Chat 幕后流程

用户浏览器 → Admin → Daemon → Actor → LLM → 回复。
⚠ 橙色高亮 = Actor 产生但客户端收不到的数据

POST /api/chat/{id}/messages
🌐 浏览器 (React)
chat.$dialogId.tsx
→
前端发送消息：用户输入文本后，先创建一条乐观消息（optimistic update）， 然后调用 sendChatMessage() POST 到 Admin。 收到 reply 后忽略返回值，重新 GET 消息历史来渲染。
↓
X-Daemon-Secret + raw body
⚙️ Admin Server :8781
proxy → daemon (sync HTTP)
→
纯代理层：不解析 chat 数据。读取请求 body → 通过同步 HTTP（线程池）转发到 Daemon → 原样返回响应。 Admin 和 Daemon 是独立进程，通过 DaemonClient.base_url (http://127.0.0.1:8780) 通信，10 秒超时。
↓
IncomingMessage + mailbox.put()
🔧 Daemon Server :8780
web_chat_message() handler
→
核心编排器：验证 secret → 解析 JSON → 找到/启动 actor → 投递消息到 mailbox → 等待回复（5 秒超时）→ 持久化消息 → 返回 JSON。 构造 IncomingMessage 包含 message_id、sender、source、content。
↓
ScheduleTriggerMessage
🤖 SimpleLoopActor
_consume_messages() loop
→
消息消费循环：后台运行 _consume_messages() 无限循环。 从 mailbox.recv() 取出消息 → 转为 ScheduleTriggerMessage → 交给 YuuAgentsActorRuntime 处理 LLM 交互 → 结果放入 turn_results 队列。
↓
🧠 Assembly + LLM
build_agent_definition()
→
大脑组装：每次 Actor 启动时创建 yuuagents Stage + Agent + LLM Client + Trace Observer。 LLM 客户端包裹在 PricingAwareLlmClient 中，自动计算费用。 费用数据写入 trace 数据库，不返回给浏览器。
↓
yext.* RPC → bridge → invoke
📦 Facade (yext)
Generated Python package
⇄
🔌 Integrations
capability.invoke()
→
集成桥梁：每次 Actor 启动时根据 allowed_capability_ids 动态生成 Python 包 yext/。 当 LLM 调用工具（如 yext.discord.send_message()），通过 TCP RPC 向 Daemon 的 IntegrationInvokeBridge 发送请求， 再委托给真正的集成实现。调用前强制检查权限白名单。
⚠ Actor 产生但客户端收不到的数据
数据维度 Actor 内部产生 客户端实际收到
文本回复 ✅ 可见 agent.history 最后一条 assistant message ✅ 收到 data.reply（纯文本）
工具调用 ✅ 可见 run_agent_loop() 中 yext.* 调用、Python 执行 ❌ 不可见 只返回最终文本，工具调用轨迹丢失
Trace ID ✅ 可见 eventbus.scope("agent.turn", {agent_id, ...}) ❌ 不可见 客户端无法关联到 trace 视图
Token / 费用 ✅ 可见 PricingAwareLlmClient 计算 input/output cost ❌ 不可见 费用数据写入 trace store，不返回
后台任务 ✅ 可见 FacadeBackgroundTaskStarted/Ended ❌ 不可见 后台任务状态不通知客户端
对话历史长度 ✅ 可见 turn_result.history_length ❌ 不可见 history_length 在服务端丢弃
集成调用结果 ✅ 可见 capability.invoke() 返回值 ❌ 不可见 只有 LLM 能处理，客户端看不到
